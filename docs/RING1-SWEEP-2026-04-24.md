## Why two sweeps — signal corruption in sweep 1

The first sweep showed: bowling × devstral 7/10, roman × devstral 4/10.
Investigating the failing workspaces revealed that devstral consistently
wrote an extra `{kata}.test.ts` at the workspace root in addition to
the implementation — 20/20 runs. qwen3 never did.

When devstral's bonus test file had a compile error, the runner flagged
`filesFailed>0` and marked the run `passed=false` even though all 7 of
the kata's real tests passed. Bowling's 70% was entirely this noise.

**Two tightenings to fix signal integrity (commit `93f30ea`):**

1. **Prompt:** explicit rule in `ollama.ts` system prompt —
   *"When tests are provided, do NOT write any new test files. Only
   implement the source files needed to make the provided tests pass.
   The provided tests ARE the contract."*
2. **Runner:** `runTests` now passes `"tests/"` to vitest when the kata
   provided tests — the model's extras (if any leak through) at
   workspace root are never executed.

Post-tightening, devstral stopped writing extra tests entirely, AND the
runner scopes to the kata's tests, so both belt and suspenders.

## Failure mode analysis (roman × devstral × 3 fails)

**Run 2 (9/10 fail):** `fromRoman('IIII')` and `fromRoman('ABC')` should
throw. Devstral's `fromRoman` has no input validation. Classic
contract-reading miss — the test spells out what "invalid" means, the
model just didn't internalize the `throws on invalid input` test as a
hard constraint.

**Run 7 (6/10 fail):** Over-corrected. Devstral added a validation
regex `/^M*(CM|CD|XC|XL|IX|IV|I)*$/` that rejects valid roman numerals
(accepts M, I, and subtractive pairs, but not freestanding V/X/L/C/D).
"Round-trips with toRoman" and several other tests fail because valid
input gets rejected. And yet `IIII` still passes the regex because
`M*I*` accepts it. The model knew it *should* validate, but botched the
pattern.

**Run 10 (9/10 fail):** Same as Run 2 — no validation at all.

**Pattern:** Devstral's `fromRoman` validation is inconsistent. 3/10
runs miss the contract entirely, 7/10 get it right. qwen3 gets it right
10/10. This is exactly what Ring 1 is for — catching which contracts
this model reads reliably.

## What this tells us about the models at Ring 1 backend

- **qwen3-coder-next is rock solid.** 30/30 across both sweeps. Clean
  signal. Fast enough at ~16s/run. Good candidate for a reference model.
- **devstral-dojo is fast and mostly correct.** 27/30 total. Twice as
  fast as qwen3 on these katas. Weakness: inconsistent at reading
  "validate input" contracts expressed only through test assertions.
- **The harness is stable.** 80 consecutive runs, zero harness errors
  across two sweeps. The shape-A and shape-B parser paths handle both
  models' natural output styles.

# Fullstack Ring 1 Sweep — 2026-04-24

Fullstack-todo-app Ring 1 sweep with the direct-Ollama harness and the
blind-oracle discipline: kata ships pre-written E2E tests under
`katas/fullstack-todo-app/e2e/`, the dev Cloove remains blind to them
during implementation, and the oracle is copied in only after the dev
phase reports done.

## Setup

- **Branch:** `rizzo/remove-cline-ollama-harness` @ `e91dc68`
- **Pipeline:** `run-fullstack.ts` — 9 phases (scaffold → dev → dev-server
  → compliance → copy blind oracle → SDET skip → E2E → unit → stop)
- **Prompt:** `prompts/fullstack-dev-v1/implement.md` (with testid
  templating + assert-conditional guardrails from the Apr-22 failure
  analysis)
- **Oracle:** `katas/fullstack-todo-app/e2e/todo-app.e2e.test.ts` (Puppeteer)
- **Models:** qwen3-coder-next:latest, devstral-dojo:latest
- **Runs:** 3 per model — 6 total. Fullstack runs are 1–2 minutes each.
- **Log:** `.runs/ring1-fullstack-sweep-20260424-141711.log`

**Headline:** devstral **beats** qwen3 on the blind oracle. 3/3 vs 0/3.
This is the opposite of the intuition going in — qwen3-coder-next is

this kata, at roughly half the dev-phase time.

## Layered findings

- **Harness is stable.** 6 end-to-end fullstack runs through the
  Ollama adapter, every dev phase completed cleanly. No parser errors,
  no harness timeouts.
- **Compliance ceiling at 95/100 is kata/inspector-side, not model.**
  Both models lose the same 5 points every run on the `clear-completed`
  element. The contract marks it `assert: conditional`
  (`visible_when: At least one todo is completed`), which means
  v-if'd out on initial render. The compliance inspector scans initial
  render and flags it missing. Documented in ring1-closure-2026-04-22.md.
  Worth fixing in the inspector, not in the model.
- **Unit tests fail 6/6 across both models.** Same pattern as
  Apr-23: models write `wrapper.trigger('toggle')` when the test
  should be `expect(wrapper.emitted('toggle')).toBeTruthy()`. Vue Test
  Utils idiom mistake — the models don't distinguish between DOM events
  and component emits. Real Ring 1 signal that *unit tests written by
  the model implementing the code tell us little about contract
  fidelity* — exactly why the blind E2E oracle matters.
- **Cross-model dev-time gap is striking.** qwen3 averaged 59s, devstral
  averaged 30s — despite devstral closing the contract better. On this
  workload, larger model ≠ better model.

## What caught the qwen3 E2E failure

Inspection of qwen3's workspaces (preserved at `/tmp/dojo-fullstack-*/`)
shows qwen3 used `todo-item-${index}` on the outer `<li>` (correct per
contract's `testid_pattern`). So the Apr-22 smoking gun (`todo.id` vs
`index`) is not the story this time. The E2E failure mode needs closer
investigation — likely a subtler contract-reading miss in interaction
handlers (toggle/delete/clear) or initial render. A follow-up dive is
worth it since qwen3 has been our "reference model" for Ring 1.

## Bugs surfaced by this sweep

1. **Dev-server orphans hang the runner.** Each fullstack run spawns a
   `vite` dev-server for compliance + E2E and doesn't terminate it.
   After qwen3's 3 runs, 3 vite servers were still alive, blocking
   `run-kata.ts` from exiting. Sweep hung for ~7 minutes until I killed
   them manually, then devstral batch proceeded normally. Filed as
   task #24. Likely a 1-line fix in `startDevServer`'s cleanup handler
   (`process.kill(-pid, 'SIGKILL')` vs `SIGTERM` to just the parent).
2. **Results JSONs don't bucket by model.** Each `run-NN.json` is keyed
   by run number only, so devstral's run-01 overwrote qwen3's run-01 on
   disk. Log file has full data but the JSONs under `results/` are
   devstral-only. Low priority fix; path needs
   `results/YYYY-MM-DD/<kata>/<prompt>/<model>/run-NN.json`.

## What this tells us about Ring 1 overall

Ring 1 now gives cross-model signal across **three** katas + **two**
models + a blind-oracle fullstack. The signal has three dimensions:

- **Contract fidelity** — roman × devstral, fullstack × qwen3 both
  show real misses
- **Speed** — devstral 2–4x faster on both backend and fullstack dev
  phases
- **Self-test quality** — neither model writes trustworthy Vue Test
  Utils unit tests; this alone justifies Ring 1's blind-oracle approach

Qwen3 is the right "Ring 1 baseline" for katas where contract is
straightforward. Devstral is the right choice when speed matters and
contracts are literal. Neither dominates the other — which is useful
news for Ring 2 (where we'll have even less control over what model
produces in test-writing phase).

## The chain of fixes that got us here

Starting point: Devstral fullstack 0/5 Overall PASS, roman 7/10,
compliance 95/100 ceiling, unit tests 0/N everywhere, dev-server
orphans hanging the runner between runs.

1. **Dev-server process-group cleanup.** `startDevServer` spawns with
   `detached: true`; `killProcessTree` signals `-pid` (whole group) on
   shutdown. No more orphaned vite/esbuild workers. Multi-run sweeps
   terminate cleanly.

2. **Compliance inspector: respect conditional visibility.**
   `runner/compliance.ts` skips elements with `assert: conditional`
   from the initial-render scan — they're expected to be absent until
   their `visible_when` precondition is met. Fixes the 95/100 ceiling.

3. **Overall-PASS gate: drop unit tests.** Unit tests are self-written
   by the dev Cloove; they're the same model grading its own work.
   Blind E2E is the real contract-fidelity signal. Unit stays
   informational, not a gate.

4. **Per-kata temperature.** Backend (bowling, roman) sets
   `harness.temperature: 0.0` in kata.yaml → deterministic. Fullstack
   keeps 0.2 for creative latitude on component design. Plumbed via
   `FireOllamaOptions.temperature` and `FullstackKataOptions.temperature`.

5. **toThrow-as-structural-validation rule.** Added to ollama.ts
   system prompt: if a test uses `expect(fn).toThrow()` on a value
   that would pass loose type/range checks, implementation MUST use
   structural validation (regex/whitelist/grammar). Pushes roman to
   10/10.

6. **Emit-pattern architectural guideline (fullstack).** Added to
   `fullstack-dev-v1` prompt: child components that interact with
   parent state MUST emit events, never mutate props or hold local
   copies. Before this: devstral stochastically alternated between
   emit-pattern (E2E green) and local-handler pattern (E2E fail on
   toggle/delete flows). After: 5/5 fullstack runs green.
