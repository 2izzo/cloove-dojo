## The numbers are lying (in a tractable way)

The runner marks a run `passed=false` when **any** test file in the
workspace fails — including files the *model wrote itself*. Devstral
consistently writes an extra `{kata}.test.ts` at the workspace root in
addition to the kata's tests/ dir. qwen3 never does. This means:

**Bowling × devstral true correctness: 10/10.** All 10 runs have the
kata's 7 tests green. The "3 failures" are runs where devstral's own
extra test file had a compile/import error, making the runner report
`passed=false` even though the kata's 7 tests all passed (7/7 visible in
`tests_passing`, `filesFailed=1` under the hood).

**Roman × devstral true correctness: 4/10.** The 6 "failures" all show
`tests_passing=18/tests_total=20` — devstral's 10 extra tests pass plus
8 of the kata's 10 pass. The 2 failing kata tests are both about input
validation: `fromRoman('IIII')` and `fromRoman('ABC')` should throw but
devstral's implementation doesn't validate. Real contract-reading miss.

## What this tells us

1. **qwen3-coder-next:latest is rock solid on backend Ring 1.** 20/20
   across both katas, no signal corruption. Confidence to use as the
   "conductor's oracle" baseline.
2. **devstral-dojo:latest has two separable issues:**
   - **Meta-compliance:** writes extra test files even when told not to.
     Fixable by tightening the system prompt ("do not write any new test
     files").
   - **Contract fidelity:** misses input validation on roman-numerals.
     This is real Ring 1 signal — exactly what the ring is supposed to
     catch. Not a harness problem.
3. **Ring 1 signal integrity depends on filtering to kata-provided test
   files.** As-is, model-written test files inflate or corrupt the
   `tests_passing` / `tests_total` count. Two options: (a) prompt the
   model to never write tests at Ring 1, (b) have the runner execute
   only `tests/*.test.ts` (kata's subdir), not the workspace root.

## Follow-up work

1. **Tighten system prompt:** add an explicit rule forbidding new test
   files at Ring 1. One line in `runner/adapters/ollama.ts` system
   prompt. Expected effect: devstral × bowling goes to 10/10 visible,
   roman stays at 4/10 visible (the real miss is real).
2. **Filter runTests to `tests/` subdir only.** Robust fix — doesn't
   trust the model to self-police. Update `runner/run-kata.ts`'s
   `runTests` to invoke `vitest run tests/` explicitly.
3. **Roman-numerals contract clarification.** The kata's description
   doesn't explicitly require `fromRoman` to throw on invalid input —
   the tests do. Two reasonable readings. Worth deciding whether
   description.md should spell out the validation requirement, or
   whether the tests-as-contract stance means we accept it as-is.
4. **Bowling-game tests leak into the workspace root.** The kata's
   `tests/bowling.test.ts` uses `import { Game } from '../bowling'` —
   implicit file-layout contract that `createWorkspace()` now honors
   via tests/ subdir. Good. Worth documenting this layout expectation
   in architecture.md.
